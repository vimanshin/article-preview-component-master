## Шаг 2 — HTML и семантика

Проверь каждый HTML-файл:

**Структура**
- Один `<h1>`, заголовки без пропуска уровней
- Правильные landmark-теги: `<header>`, `<nav>`, `<main>`, `<section>`,
  `<footer>` — не лишние `<div>` там, где есть семантический тег
- `lang` на `<html>`

**Доступность**
- У каждого `<img>` осмысленный `alt` (у декоративных — `alt=""`)
- `<label>` связан с `<input>` через `for`/`id`
- Иконки без текста имеют `aria-label`
- Декоративные SVG имеют `aria-hidden="true"`
- Интерактивные раскрывашки (меню, аккордеон) — `aria-expanded`,
  `aria-controls`
- Активный пункт навигации — `aria-current="page"`
- Группы радиокнопок в `<fieldset>` + `<legend>`
- `<button>` для действий, `<a>` для навигации — не наоборот
- `tabindex` не больше 0

**Атрибуты**
- Правильные `type` у `<input>`: email, tel, number, password
- `required` у обязательных полей

## Шаг 3 — SCSS и качество кода

**Модульность**
- Используется `@use`, не `@import`
- Обращения к переменным через namespace: `variables.$name`,
  `mixins.tablet()` — не напрямую
- Нет `@use ... as *`

**BEM и вложенность**
- BEM-нотация: `.block`, `.block__element`, `.block--modifier`
- Вложенность не глубже 2–3 уровней
- `&__element` и `&--modifier` — через `&`, не дублируя имя блока

**Mobile-first**
- Базовые стили без медиа-запросов (мобильные)
- Расширения через `min-width`, не `max-width`
- Брейкпоинты через миксины, не хардкод цифр

**Отступы между соседними элементами**
- Основной паттерн — **`margin-top`** у следующего элемента, а не
  `margin-bottom` у предыдущего
- Первый элемент в потоке не должен получать лишний `margin-top` (или он
  обнулён через `:first-child` / `:first-of-type`)
- Одинаковые отступы во flex/grid — через `gap`; разные расстояния —
  через `margin-top` у соответствующих элементов
- `margin-bottom` как дефолт — замечание в review; допустим только как
  исключение (последний элемент, схлопывание margin и т.п.)
- При нарушении — укажи файл, селектор и предложи замену на `margin-top`

**Переменные**
- Переменные только для повторяющихся или семантических значений
- Разовые самоочевидные значения — хардкодом

**Избыточность и оптимизация**
- Дублирующиеся свойства с одинаковым значением — объединить
- Свойства, которые перекрывают друг друга без причины — убрать лишнее
- Можно ли сократить через shorthand:
  `margin: 0 auto` вместо двух отдельных, `border: 1px solid #ccc` и т.д.
- Селекторы, которые можно вынести наверх как общие стили
- Неиспользуемые переменные, классы, закомментированный мёртвый код

**Порядок свойств внутри блока** (стандарт индустрии)
Проверь соответствие порядку: позиционирование → блочная модель →
типографика → оформление → прочее:

```
position / top / right / bottom / left / z-index
display / flex-* / grid-* / gap
width / height / min-* / max-*
margin / padding
font-* / line-height / text-* / color
background / border / border-radius / box-shadow / outline
opacity / transform / transition / animation
cursor / pointer-events / overflow
```

Если порядок нарушен — покажи конкретные файлы и блоки.

## Шаг 4 — JavaScript и качество кода

**Переменные и именование**
- `const` по умолчанию, `let` только там, где значение переприсваивается,
  `var` отсутствует
- Обработчики событий названы с префиксом `handle`
- Булевы переменные/функции — с `is`/`has`/`should`

**Структура**
- Ранний return вместо вложенных `if`
- Одна функция — одна ответственность

**Работа с DOM**
- Перед обращением к элементу — проверка, что он существует (`if (!el) return`)
- JS-хуки отделены от стилевых классов (`data-js`/`.js-`, а не переиспользование
  BEM-классов из SCSS)
- `textContent` для текста; `innerHTML` не используется с недоверенными данными

**События и доступность**
- `addEventListener`, а не `onclick=""` в HTML
- Делегирование событий для списков/динамических элементов
- Кастомные интерактивные элементы обрабатывают клавиатуру (`Enter`/`Space`),
  не только `click`
- Состояние синхронизировано с ARIA (`aria-expanded` и т.п. обновляются в JS)

**Прочее**
- Строгое сравнение `===`/`!==`
- Нет забытых `console.log`, закомментированного мёртвого кода

## Шаг 5 — Соответствие дизайн-макету

Сравни по каждой секции:

| Параметр | Проверяем |
|---|---|
| Отступы | padding, margin, gap совпадают с макетом |
| Цвета | соответствуют переменным из `_variables.scss` |
| Типографика | font-size, font-weight, line-height, letter-spacing |
| Размеры | ширины блоков, изображений, иконок |
| Mobile | мобильная версия соответствует мобильному фрейму |
| Tablet | планшетная версия соответствует планшетному фрейму |
| Desktop | десктоп соответствует десктопному фрейму |
| Состояния | есть ли hover, focus, active там, где они в макете |

## Шаг 6 — Баги и потенциальные проблемы

- Незакрытые теги
- Сломанные пути к изображениям, шрифтам, файлам
- Конфликтующие стили (одно свойство переопределяется несколько раз)
- Элементы, которые могут вылезать за контейнер (нет `overflow: hidden`)
- Фиксированные `px`-ширины, которые сломают мобильную версию
- Изображения без `max-width: 100%`
- `z-index` конфликты
- Обращение к DOM-элементу без проверки на `null` (упадёт, если элемента нет)
- `innerHTML` с данными, которые могут прийти от пользователя (XSS)
- Дублирующиеся обработчики событий (навешаны повторно без снятия старых)
